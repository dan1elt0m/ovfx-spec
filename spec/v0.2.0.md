# OVFX — Open Visual Field eXchange

**Specification version:** 0.2.0 (draft)
**Status:** Public draft — subject to change before 1.0.0
**Date:** 2026-04-15
**Supersedes:** 0.1.0 (in pre-adoption phase)

## Abstract

This document defines **OVFX (Open Visual Field eXchange)**, a JSON-based interchange format for visual-field perimetry test results. OVFX captures the recorded points, the test-time setup, the stimulus definitions, and the test metadata for kinetic (Goldmann), static, and ring/arc perimetry across any hardware modality — consumer screen, clinical bowl perimeter, VR headset, or other. It is designed for use by self-test apps, clinical software, research tools, and longitudinal personal archives.

## 1. Conventions

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**, **SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY**, and **OPTIONAL** in this document are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) when, and only when, they appear in all capitals.

JSON terminology follows [RFC 8259](https://www.rfc-editor.org/rfc/rfc8259). All field names are case-sensitive.

## 2. Scope and goals

OVFX exists to make visual-field perimetry data portable. The format is intentionally:

- **Small** — a complete result fits in a few kilobytes of JSON.
- **Human-readable** — every field name is descriptive; no opaque codes.
- **Device-agnostic** — the core data model (points in polar degrees) is identical for a screen-based self-test, a clinical bowl perimeter, and a VR headset.
- **Setup-aware** — the physical setup that produced the data is documented under a discriminated `setup` object, so a screen-based consumer can reproduce a screen-based test at 1:1 scale without contaminating the data model of bowl or VR producers.
- **Forward-compatible** — extensions are namespaced and unknown fields MUST be preserved by tools that round-trip the document.
- **Privacy-preserving by default** — no protected health information (PHI) is required.

OVFX is **not** a clinical exchange format. It is not a replacement for DICOM Ophthalmic Visual Field Static Perimetry Measurements (Supplement 80) and does not attempt to encode the rich clinician-facing context that DICOM provides. OVFX targets the gap between vendor lock-in and DICOM's complexity.

## 3. Document model

An OVFX document is a single JSON object representing **one test result for one eye on one date**. A document representing both eyes of a binocular test SHOULD be split into two documents (one per eye) sharing a `binocularGroup` identifier, OR a single document with `test.eye` set to `"both"` and a single combined point set.

### 3.1 Top-level fields

| Field          | Type   | Required | Description                                                                           |
| -------------- | ------ | -------- | ------------------------------------------------------------------------------------- |
| `ovfxVersion`  | string | yes      | The semantic version of this specification the document conforms to (e.g. `"0.2.0"`). |
| `id`           | string | yes      | A globally unique identifier for this result. UUIDv4 is **RECOMMENDED**.              |
| `createdAt`    | string | yes      | RFC 3339 timestamp of when the test was performed (not when the file was written).    |
| `subject`      | object | no       | Optional subject metadata. See §3.2.                                                  |
| `test`         | object | yes      | Test descriptor. See §3.3.                                                            |
| `calibration`  | object | yes      | Test-time setup and user calibration. See §3.4.                                       |
| `stimuli`      | array  | yes      | The stimulus definitions used during the test. See §3.5.                              |
| `points`       | array  | yes      | The recorded test points. See §3.6.                                                   |
| `isopters`     | array  | no       | Optional precomputed isopter areas. Always derivable from `points`. See §3.7.         |
| `software`     | object | no       | Identification of the software that produced the document. See §3.8.                  |
| `extensions`   | object | no       | Implementation-defined extension fields. See §6.                                      |

A document MUST contain `ovfxVersion`, `id`, `createdAt`, `test`, `calibration`, `stimuli`, and `points`. All other top-level fields are OPTIONAL.

### 3.2 `subject`

The `subject` object is OPTIONAL. When present, it MAY contain only the fields below. Implementations **MUST NOT** include personal identifying information (name, date of birth, address, identifiers used to look up clinical records) in this object.

| Field         | Type   | Required | Description                                                                  |
| ------------- | ------ | -------- | ---------------------------------------------------------------------------- |
| `anonymousId` | string | no       | A subject identifier scoped to the producing application (e.g. a device ID). |
| `birthYear`   | number | no       | Birth year as an integer (e.g. `1989`). Coarser than birth date.             |
| `sex`         | string | no       | One of `"male"`, `"female"`, `"other"`, `"unknown"`.                          |

Producers SHOULD omit this object entirely unless the receiving application has a documented need.

### 3.3 `test`

| Field            | Type    | Required | Description                                                                       |
| ---------------- | ------- | -------- | --------------------------------------------------------------------------------- |
| `type`           | string  | yes      | One of `"kinetic"`, `"static"`, `"ring"`. See §4.                                |
| `eye`            | string  | yes      | One of `"left"`, `"right"`, `"both"`.                                             |
| `extendedField`  | boolean | no       | `true` if the test extended into the periphery beyond the standard testable area. |
| `binocularGroup` | string  | no       | Identifier shared by two single-eye documents from the same binocular session.    |
| `notes`          | string  | no       | Free-form human-readable notes (≤ 2000 characters).                               |

### 3.4 `calibration`

The `calibration` object holds both device-agnostic measurements of the test (e.g. `maxEccentricityDeg`) and a discriminated `setup` sub-object describing the specific hardware used. This split keeps the core data model portable: a consumer that only cares about angular positions never has to understand the setup.

| Field                | Type   | Required | Description                                                                                                |
| -------------------- | ------ | -------- | ---------------------------------------------------------------------------------------------------------- |
| `maxEccentricityDeg` | number | yes      | The maximum eccentricity reached by the test, in degrees of visual angle.                                  |
| `reactionTimeMs`     | number | no       | The user's measured reaction time in milliseconds, used for kinetic-perimetry compensation.                |
| `setup`              | object | yes      | Discriminated setup sub-object. Always has a `type` field; may have a matching device-specific sub-object. |

#### 3.4.1 `setup`

| Field     | Type   | Required | Description                                                                         |
| --------- | ------ | -------- | ----------------------------------------------------------------------------------- |
| `type`    | string | yes      | One of `"screen"`, `"bowl"`, `"vr"`, `"other"`. See §3.4.2–§3.4.5.                  |
| `screen`  | object | no       | REQUIRED when `type === "screen"`; MUST be absent otherwise. See §3.4.2.            |
| `bowl`    | object | no       | RECOMMENDED when `type === "bowl"`. See §3.4.3.                                     |
| `vr`      | object | no       | RECOMMENDED when `type === "vr"`. See §3.4.4.                                       |
| `other`   | object | no       | When `type === "other"`, an implementation-defined object describing the setup.      |

#### 3.4.2 `setup.screen`

Used for tests performed on a consumer display (desktop monitor, laptop, tablet, phone).

| Field                | Type   | Required | Description                                                                                            |
| -------------------- | ------ | -------- | ------------------------------------------------------------------------------------------------------ |
| `viewingDistanceCm`  | number | yes      | The viewing distance from the eye to the screen during the test, in centimetres.                       |
| `pixelsPerDegree`    | number | yes      | Screen pixels subtended by 1° of visual angle at `viewingDistanceCm`. Canonical conversion factor.     |
| `screenWidthPx`      | number | yes      | The width of the screen used for the test, in pixels.                                                  |
| `screenHeightPx`     | number | yes      | The height of the screen used for the test, in pixels.                                                 |
| `fixationOffsetPx`   | number | yes      | Horizontal offset of the fixation point from screen centre, in pixels. Negative = left.                |
| `brightnessFloor`    | number | no       | The minimum visible stimulus opacity (0–1) measured during calibration.                                |

`pixelsPerDegree` MUST be the value derived from the calibration step at test time, not back-computed from individual points.

#### 3.4.3 `setup.bowl`

Used for tests performed on a clinical bowl perimeter (Goldmann, Octopus, Humphrey).

| Field                     | Type   | Required | Description                                                                 |
| ------------------------- | ------ | -------- | --------------------------------------------------------------------------- |
| `bowlRadiusCm`            | number | no       | Radius of the bowl in centimetres (Goldmann standard is 33 cm).             |
| `backgroundLuminanceCdM2` | number | no       | Background luminance of the bowl, in candelas per square metre.             |
| `deviceModel`             | string | no       | Free-form manufacturer + model string (e.g. `"Haag-Streit Goldmann 940ST"`). |

#### 3.4.4 `setup.vr`

Used for tests performed on a virtual-reality headset.

| Field                | Type   | Required | Description                                                                         |
| -------------------- | ------ | -------- | ----------------------------------------------------------------------------------- |
| `headset`            | string | no       | Free-form headset name (e.g. `"Meta Quest 3"`).                                    |
| `fieldOfViewDeg`     | number | no       | Nominal horizontal field of view of the headset, in degrees.                        |
| `pixelsPerDegree`    | number | no       | Effective pixel density at the centre of the lens, in pixels per degree.            |

#### 3.4.5 `setup.other`

Any unclassified setup. The contents are implementation-defined but MUST be a JSON object.

### 3.5 `stimuli`

`stimuli` is an array of stimulus definitions used in the test. At least one entry is REQUIRED.

| Field       | Type   | Required | Description                                                                                    |
| ----------- | ------ | -------- | ---------------------------------------------------------------------------------------------- |
| `key`       | string | yes      | A short identifier, unique within the document. Goldmann notation (`"V4e"`, …) is RECOMMENDED. |
| `sizeDeg`   | number | yes      | Stimulus angular diameter in degrees.                                                          |
| `intensity` | number | yes      | Rendered stimulus brightness as a fraction of full brightness (0–1).                           |
| `color`     | string | no       | A CSS-format colour used for visualisation, e.g. `"#60a5fa"`.                                 |
| `label`     | string | no       | Human-readable label (defaults to `key`).                                                      |

Two distinct stimuli MAY share `sizeDeg`. They MUST NOT share `key`.

### 3.6 `points`

`points` is an array of recorded test points. Order is not significant, but producers SHOULD preserve the test-time order to aid reproducibility.

| Field                | Type    | Required | Description                                                                                      |
| -------------------- | ------- | -------- | ------------------------------------------------------------------------------------------------ |
| `stimulusKey`        | string  | yes      | Reference to a `key` in the `stimuli` array.                                                    |
| `meridianDeg`        | number  | yes      | Polar angle in degrees, measured counter-clockwise from the positive x-axis. Range: `[0, 360)`. |
| `eccentricityDeg`    | number  | yes      | Distance from fixation in degrees. MUST be ≥ 0.                                                  |
| `detected`           | boolean | yes      | `true` if the user reported seeing the stimulus, `false` otherwise.                              |
| `rawEccentricityDeg` | number  | no       | The eccentricity before reaction-time compensation (kinetic perimetry only).                     |
| `responseTimeMs`     | number  | no       | The user's response time for this point, in milliseconds.                                        |
| `sectorIdx`          | number  | no       | Sector index (ring/arc tests only).                                                              |
| `repeat`             | boolean | no       | `true` if this point was an automatic retest of an earlier outlier or unseen position.            |

### 3.7 `isopters`

`isopters` is an OPTIONAL array of precomputed per-stimulus area summaries. Including this array is purely a convenience; consumers SHOULD always be able to recompute these values from `points`.

| Field         | Type   | Required | Description                                                     |
| ------------- | ------ | -------- | --------------------------------------------------------------- |
| `stimulusKey` | string | yes      | Reference to a `key` in the `stimuli` array.                    |
| `areaSqDeg`   | number | yes      | The area enclosed by the detected boundary, in square degrees.  |

### 3.8 `software`

| Field     | Type   | Required | Description                             |
| --------- | ------ | -------- | --------------------------------------- |
| `name`    | string | yes      | Producing application name.              |
| `version` | string | yes      | Producing application version.           |
| `url`     | string | no       | Homepage of the producing application.   |

## 4. Test types

### 4.1 Kinetic (`"kinetic"`)

A kinetic test moves a stimulus from outside the visual field inward along each meridian until the user signals detection. Each point typically has `detected = true`. Points with `detected = false` represent meridians where the stimulus reached the central area without being seen.

`responseTimeMs` and `rawEccentricityDeg` SHOULD be present for kinetic points when reaction-time compensation is applied.

### 4.2 Static (`"static"`)

A static test flashes individual stimuli at fixed grid positions. Each point represents one trial; `detected = false` means the stimulus was not perceived in the response window.

### 4.3 Ring (`"ring"`)

A ring/arc test maps scotoma boundaries via user-controlled arc expansion. `sectorIdx` SHOULD be present so consumers can group points by sector.

## 5. Coordinate system

OVFX uses a single canonical coordinate system:

- **Origin** is the fixation point.
- **`meridianDeg`** is the angle in degrees, measured counter-clockwise from the positive x-axis. `0°` is to the user's right, `90°` is up, `180°` is to the user's left, `270°` is down.
- **`eccentricityDeg`** is the radial distance from fixation in degrees of visual angle.
- **Eye orientation:** the data is recorded **as the user sees it**. The temporal field of the right eye is on the user's right (positive x); the temporal field of the left eye is on the user's left (negative x).

Screen-specific terms (pixels, fixation offset) are confined to `calibration.setup.screen` and never leak into `points`.

## 6. Extensions

### 6.1 Top-level `extensions` object

The top-level `extensions` object MAY contain implementation-defined fields. Each top-level key under `extensions` MUST be a reverse-DNS namespace owned by the implementer (e.g. `"com.example.myapp"`).

### 6.2 Underscore-prefixed point fields

Per-point fields whose name begins with an underscore (e.g. `"_myField"`) MUST be preserved by tools that round-trip the document and MUST NOT be interpreted by general-purpose consumers.

Tools that re-emit an OVFX document **MUST preserve** unknown top-level fields, unknown `extensions` keys, and underscore-prefixed point fields.

## 7. Conformance

A **conforming OVFX document** is a JSON document that:

1. Validates against the JSON Schema published at `schema/v0.2.0/ovfx.schema.json`.
2. Contains all REQUIRED fields enumerated in §3.
3. Uses only the test types defined in §4 (or an extension type explicitly negotiated between producer and consumer).
4. Uses the coordinate system defined in §5.
5. If `calibration.setup.type === "screen"`, includes a `calibration.setup.screen` sub-object.

A consumer **MUST** reject documents whose `ovfxVersion` is from a higher MAJOR version than it supports.

## 8. Versioning

- **MAJOR** — backward-incompatible schema change.
- **MINOR** — backward-compatible additions.
- **PATCH** — clarifications and editorial corrections.

Each spec version lives at a stable URL under `spec/<version>.md` and `schema/<version>/`. Published spec files are immutable except for editorial errata, which are recorded in `CHANGELOG.md`.

## 9. Security and privacy considerations

- OVFX documents **MUST NOT** contain PHI by default. The OPTIONAL `subject` object intentionally omits fields like name and exact birth date.
- A self-test app emitting OVFX SHOULD prompt the user before including `subject.anonymousId`.
- OVFX documents are plain JSON and are not encrypted by the format.
- The `notes` field is free-form and is the most likely place for a user to accidentally enter PHI. Implementers SHOULD warn users before allowing free-form notes.

## 10. Migration from 0.1.0

0.1.0 put `screenWidthPx`, `screenHeightPx`, `pixelsPerDegree`, `fixationOffsetPx`, `viewingDistanceCm`, and `brightnessFloor` directly under `calibration`. 0.2.0 moves them into `calibration.setup.screen` and adds `calibration.setup.type`. `maxEccentricityDeg` and `reactionTimeMs` stay at the top of `calibration`.

A 0.1.0 document can be mechanically upgraded by:

1. Creating `calibration.setup = { "type": "screen", "screen": { ... } }`.
2. Moving the six fields listed above into `calibration.setup.screen`.
3. Bumping `ovfxVersion` from `"0.1.0"` to `"0.2.0"`.

No point data changes. No producer known to exist has adopted 0.1.0 at the time of this release.
