# OVFX — Open Visual Field eXchange

**Specification version:** 0.4.0 (draft)
**Status:** Public draft — subject to change before 1.0.0
**Date:** 2026-04-17
**Supersedes:** 0.3.0

## Abstract

This document defines **OVFX (Open Visual Field eXchange)**, a JSON-based interchange format for visual-field perimetry test results. OVFX captures the recorded points, the test-time setup, the stimulus definitions, per-test reliability indicators, and the test metadata for kinetic (Goldmann), static, and ring/arc perimetry across any hardware modality — consumer screen, clinical bowl perimeter, VR headset, or other. It is designed for use by self-test apps, clinical software, research tools, and longitudinal personal archives.

## 1. Conventions

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**, **SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY**, and **OPTIONAL** in this document are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) when, and only when, they appear in all capitals.

JSON terminology follows [RFC 8259](https://www.rfc-editor.org/rfc/rfc8259). All field names are case-sensitive.

## 2. Scope and goals

OVFX exists to make visual-field perimetry data portable. The format is intentionally:

- **Small** — a complete result fits in a few kilobytes of JSON.
- **Human-readable** — every field name is descriptive; no opaque codes.
- **Device-agnostic** — the core data model (points in polar degrees) is identical for a screen-based self-test, a clinical bowl perimeter, and a VR headset.
- **Setup-aware** — the physical setup is documented under a discriminated `setup` object so device-specific fields never contaminate the core data model.
- **Clinically compatible** — reliability indicators, threshold sensitivity, and test pattern/strategy fields are supported so clinical HFA or Octopus exports can be expressed in OVFX without loss.
- **Forward-compatible** — extensions are namespaced and unknown fields MUST be preserved by tools that round-trip the document.
- **Privacy-preserving by default** — no protected health information (PHI) is required.

OVFX is **not** a clinical exchange format in the DICOM sense. It does not replace DICOM Ophthalmic Visual Field Static Perimetry Measurements (Supplement 80) and does not attempt to encode the rich clinician-facing context that DICOM provides. OVFX targets the gap between vendor lock-in and DICOM's complexity.

## 3. Document model

An OVFX document is a single JSON object representing **one test result for one eye on one date**. A document representing both eyes of a binocular test SHOULD be split into two documents (one per eye) sharing a `binocularGroup` identifier, OR a single document with `test.eye` set to `"both"` and a single combined point set.

### 3.1 Top-level fields

| Field          | Type   | Required | Description                                                                           |
| -------------- | ------ | -------- | ------------------------------------------------------------------------------------- |
| `ovfxVersion`  | string | yes      | The semantic version of this specification the document conforms to (e.g. `"0.4.0"`). |
| `id`           | string | yes      | A globally unique identifier for this result. UUIDv4 is **RECOMMENDED**.              |
| `createdAt`    | string | yes      | RFC 3339 timestamp of when the test was performed (not when the file was written).    |
| `subject`      | object | no       | Optional subject metadata. See §3.2.                                                  |
| `test`         | object | yes      | Test descriptor. See §3.3.                                                            |
| `calibration`  | object | yes      | Test-time setup and user calibration. See §3.4.                                       |
| `stimuli`      | array  | yes      | The stimulus definitions used during the test. See §3.5.                              |
| `points`       | array  | yes      | The recorded test points. See §3.6.                                                   |
| `reliability`         | object | no       | Per-test reliability indicators (false positives, fixation losses, etc.). See §3.7.   |
| `reliabilityIndices`  | object | no       | Raw catch-trial and fixation-reliability counts. See §3.7a.                           |
| `isopters`            | array  | no       | Optional precomputed isopter areas. See §3.8.                                         |
| `software`     | object | no       | Identification of the software that produced the document. See §3.9.                  |
| `extensions`   | object | no       | Implementation-defined extension fields. See §6.                                      |

A document MUST contain `ovfxVersion`, `id`, `createdAt`, `test`, `calibration`, `stimuli`, and `points`. All other top-level fields are OPTIONAL.

`reliabilityIndices` is OPTIONAL. When present, all four of its fields are REQUIRED; the derived FA% and FPRR% metrics MUST be computed at display time and MUST NOT be stored in the document.

### 3.2 `subject`

The `subject` object is OPTIONAL. When present, it MAY contain only the fields below. Implementations **MUST NOT** include personal identifying information (name, date of birth, address, identifiers used to look up clinical records) in this object.

| Field         | Type   | Required | Description                                                                  |
| ------------- | ------ | -------- | ---------------------------------------------------------------------------- |
| `anonymousId` | string | no       | A subject identifier scoped to the producing application (e.g. a device ID). |
| `birthYear`   | number | no       | Birth year as an integer (e.g. `1989`). Coarser than birth date.             |
| `sex`         | string | no       | One of `"male"`, `"female"`, `"other"`, `"unknown"`.                          |

Producers SHOULD omit this object entirely unless the receiving application has a documented need.

### 3.3 `test`

| Field             | Type    | Required | Description                                                                                        |
| ----------------- | ------- | -------- | -------------------------------------------------------------------------------------------------- |
| `type`            | string  | yes      | One of `"kinetic"`, `"static"`, `"ring"`. See §4.                                                 |
| `eye`             | string  | yes      | One of `"left"`, `"right"`, `"both"`.                                                              |
| `extendedField`   | boolean | no       | `true` if the test extended into the periphery beyond the standard testable area.                  |
| `binocularGroup`  | string  | no       | Identifier shared by two single-eye documents from the same binocular session.                     |
| `pattern`         | string  | no       | Clinical grid name — e.g. `"24-2"`, `"30-2"`, `"10-2"`, `"Esterman"`, `"Goldmann"`, `"custom"`.    |
| `strategy`        | string  | no       | Testing strategy — e.g. `"threshold"`, `"SITA-standard"`, `"SITA-fast"`, `"screening"`, `"kinetic"`, `"ring-sector"`. |
| `durationSeconds` | number  | no       | Total wall-clock duration of the test, in seconds.                                                 |
| `stimulusDurationMs`            | number  | no     | How long each stimulus was presented, in milliseconds. Clinical HFA convention is 200 ms.          |
| `interStimulusIntervalMs`       | number  | no     | Fixed minimum interval between successive stimuli, in milliseconds.                                |
| `interStimulusIntervalJitterMs` | number  | no     | Additional random interval added on top of `interStimulusIntervalMs` (uniform in `[0, jitter]`).   |
| `notes`           | string  | no       | Free-form human-readable notes (≤ 2000 characters).                                                |

`pattern` and `strategy` are intentionally free-form strings; consumers SHOULD recognise the common clinical values listed above and treat unknown values as opaque labels.

`stimulusDurationMs` and the two `interStimulusInterval*` fields describe the presentation timing of the test. They are OPTIONAL because not every producer captures them, but consumers that compare results across devices SHOULD prefer documents that include them — response-time and threshold measurements are only comparable when the underlying presentation timing is known.

### 3.4 `calibration`

The `calibration` object holds user-level measurements of the test (e.g. `reactionTimeMs`) and a discriminated `setup` sub-object describing the specific hardware used. Device-specific geometry — including the maximum eccentricity reached — lives inside `setup.<type>`, so a consumer that only understands one setup type does not have to interpret fields that belong to another.

| Field            | Type   | Required | Description                                                                                                |
| ---------------- | ------ | -------- | ---------------------------------------------------------------------------------------------------------- |
| `reactionTimeMs` | number | no       | The user's measured reaction time in milliseconds, used for kinetic-perimetry compensation.                |
| `setup`          | object | yes      | Discriminated setup sub-object. Always has a `type` field and a matching device-specific sub-object.       |

#### 3.4.1 `setup`

| Field     | Type   | Required | Description                                                                         |
| --------- | ------ | -------- | ----------------------------------------------------------------------------------- |
| `type`    | string | yes      | One of `"screen"`, `"bowl"`, `"vr"`, `"other"`. See §3.4.2–§3.4.5.                  |
| `screen`  | object | no       | REQUIRED when `type === "screen"`; MUST be absent otherwise. See §3.4.2.            |
| `bowl`    | object | no       | RECOMMENDED when `type === "bowl"`. See §3.4.3.                                     |
| `vr`      | object | no       | RECOMMENDED when `type === "vr"`. See §3.4.4.                                       |
| `other`   | object | no       | When `type === "other"`, an implementation-defined object describing the setup.      |

#### 3.4.2 `setup.screen`

Used for tests performed on a consumer display (desktop monitor, laptop, tablet, phone).

| Field                | Type   | Required | Description                                                                                            |
| -------------------- | ------ | -------- | ------------------------------------------------------------------------------------------------------ |
| `viewingDistanceCm`  | number | yes      | The viewing distance from the eye to the screen during the test, in centimetres.                       |
| `pixelsPerDegree`    | number | yes      | Screen pixels subtended by 1° of visual angle at `viewingDistanceCm`. Canonical conversion factor.     |
| `screenWidthPx`      | number | yes      | The width of the screen used for the test, in pixels.                                                  |
| `screenHeightPx`     | number | yes      | The height of the screen used for the test, in pixels.                                                 |
| `fixationOffsetPx`   | number | yes      | Horizontal offset of the fixation point from screen centre, in pixels. Negative = left.                |
| `maxEccentricityDeg` | number | yes      | The maximum eccentricity reached by the test on this screen, in degrees.                               |
| `brightnessFloor`    | number | no       | The minimum visible stimulus opacity (0–1) measured during calibration.                                |
| `maxLuminanceCdM2`        | number | no  | Absolute luminance of a full-brightness (`intensity = 1.0`) stimulus on this screen, in cd/m². Defines the `0 dB` reference for `points[].sensitivityDb`. |
| `backgroundLuminanceCdM2` | number | no  | Background luminance of the screen during the test, in cd/m².                                          |

`pixelsPerDegree` MUST be the value derived from the calibration step at test time, not back-computed from individual points.

`maxLuminanceCdM2` and `backgroundLuminanceCdM2` are OPTIONAL because not every consumer-screen self-test has a calibrated photometer. Producers that have run a photometric calibration (e.g. using a published gamma profile or a hardware sensor) SHOULD include these fields — they are the only way to make `points[].sensitivityDb` values portable across devices.

#### 3.4.3 `setup.bowl`

Used for tests performed on a clinical bowl perimeter (Goldmann, Octopus, Humphrey).

| Field                     | Type   | Required | Description                                                                 |
| ------------------------- | ------ | -------- | --------------------------------------------------------------------------- |
| `maxEccentricityDeg`      | number | yes      | The maximum eccentricity reached by the test in the bowl, in degrees.       |
| `bowlRadiusCm`            | number | no       | Radius of the bowl in centimetres (Goldmann standard is 33 cm).             |
| `backgroundLuminanceCdM2` | number | no       | Background luminance of the bowl, in candelas per square metre.             |
| `deviceModel`             | string | no       | Free-form manufacturer + model string (e.g. `"Haag-Streit Goldmann 940ST"`). |

#### 3.4.4 `setup.vr`

Used for tests performed on a virtual-reality headset.

| Field                | Type   | Required | Description                                                                         |
| -------------------- | ------ | -------- | ----------------------------------------------------------------------------------- |
| `maxEccentricityDeg` | number | yes      | The maximum eccentricity reached by the test in the headset, in degrees.            |
| `headset`            | string | no       | Free-form headset name (e.g. `"Meta Quest 3"`).                                    |
| `fieldOfViewDeg`     | number | no       | Nominal horizontal field of view of the headset, in degrees.                        |
| `pixelsPerDegree`    | number | no       | Effective pixel density at the centre of the lens, in pixels per degree.            |

#### 3.4.5 `setup.other`

Any unclassified setup. The contents are implementation-defined but MUST be a JSON object and SHOULD include a `maxEccentricityDeg` field for consistency with the typed setup sub-objects.

### 3.5 `stimuli`

`stimuli` is an array of stimulus definitions used in the test. At least one entry is REQUIRED.

| Field       | Type   | Required | Description                                                                                    |
| ----------- | ------ | -------- | ---------------------------------------------------------------------------------------------- |
| `key`           | string | yes      | A short identifier, unique within the document. Goldmann notation (`"V4e"`, …) is RECOMMENDED. |
| `sizeDeg`       | number | yes      | Stimulus angular diameter in degrees.                                                          |
| `intensity`     | number | yes      | Rendered stimulus brightness as a fraction of full brightness (0–1).                           |
| `luminanceCdM2` | number | no       | Absolute luminance of this stimulus when presented, in cd/m². Enables cross-device comparison.  |
| `color`         | string | no       | A CSS-format colour used for visualisation, e.g. `"#60a5fa"`.                                 |
| `label`         | string | no       | Human-readable label (defaults to `key`).                                                      |

Two distinct stimuli MAY share `sizeDeg`. They MUST NOT share `key`.

`luminanceCdM2` is OPTIONAL. When present, it represents the measured or calibrated absolute luminance at which the stimulus is rendered, independent of the device's full-brightness ceiling. A consumer can derive `luminanceCdM2` from `intensity` × `calibration.setup.screen.maxLuminanceCdM2` (or the analogous bowl field) when it is absent, but the explicit value SHOULD be preferred when available.

### 3.6 `points`

`points` is an array of recorded test points. Order is not significant, but producers SHOULD preserve the test-time order.

| Field                | Type    | Required | Description                                                                                      |
| -------------------- | ------- | -------- | ------------------------------------------------------------------------------------------------ |
| `stimulusKey`        | string  | yes      | Reference to a `key` in the `stimuli` array.                                                    |
| `meridianDeg`        | number  | yes      | Polar angle in degrees, counter-clockwise from the positive x-axis. Range: `[0, 360)`.           |
| `eccentricityDeg`    | number  | yes      | Distance from fixation in degrees. MUST be ≥ 0.                                                  |
| `detected`           | boolean | yes      | `true` if the user reported seeing the stimulus, `false` otherwise.                              |
| `sensitivityDb`      | number  | no       | Threshold sensitivity at this location, in decibels. Relevant for static threshold perimetry.    |
| `rawEccentricityDeg` | number  | no       | The eccentricity before reaction-time compensation (kinetic perimetry only).                     |
| `responseTimeMs`     | number  | no       | The user's response time for this point, in milliseconds.                                        |
| `sectorIdx`          | number  | no       | Sector index (ring/arc tests only).                                                              |
| `repeat`             | boolean | no       | `true` if this point was an automatic retest of an earlier outlier or unseen position.            |
| `catchTrial`         | boolean | no       | `true` if this point was a blindspot catch trial, not a real sensitivity probe. See §3.6a.       |

`sensitivityDb` follows the perimetry convention: larger numbers mean a more sensitive eye (a lower threshold intensity). `0 dB` corresponds to the maximum stimulus luminance of the producing hardware. When `sensitivityDb` is present for a static-perimetry point, `detected` is typically `true`; a consumer MUST treat the two fields as independent observations.

### 3.6a `points[].catchTrial`

`catchTrial` is an OPTIONAL boolean field on each point. When `true`, the point represents a **blindspot catch trial** — a stimulus presented at the subject's physiological blind spot to probe fixation stability — rather than a real sensitivity probe.

Rules:

- Catch-trial points **MUST be excluded** from any aggregate sensitivity, area, or isopter computation by downstream consumers.
- A catch trial where `detected: true` indicates a **fixation loss** (the subject was not fixating).
- When absent or `false`, the point is a normal sensitivity probe.
- Catch trials MAY use any `stimulusKey` defined in the `stimuli` array; the key does not need to differ from the probes.

**Example** (two catch-trial points embedded in a static perimetry point array):

```json
{ "stimulusKey": "V4e", "meridianDeg": 15,  "eccentricityDeg": 65.0, "detected": true  },
{ "stimulusKey": "V4e", "meridianDeg": 195, "eccentricityDeg": 14.5, "detected": false, "catchTrial": true },
{ "stimulusKey": "V4e", "meridianDeg": 195, "eccentricityDeg": 14.5, "detected": true,  "catchTrial": true }
```

The two points with `"catchTrial": true` above are blindspot trials; `detected: true` on the second one signals a fixation loss at that presentation.

### 3.7 `reliability`

Per-test quality metrics, mirroring the classical HFA reliability triad. All fields are OPTIONAL; producers SHOULD include them when the test measured them.

| Field               | Type   | Required | Description                                                                               |
| ------------------- | ------ | -------- | ----------------------------------------------------------------------------------------- |
| `falsePositiveRate` | number | no       | Catch-trial false-positive rate in `[0, 1]`. Higher means the user pressed without seeing. |
| `falseNegativeRate` | number | no       | Catch-trial false-negative rate in `[0, 1]`. Higher means the user missed previously-seen stimuli. |
| `fixationLossRate`  | number | no       | Blind-spot fixation-loss rate in `[0, 1]`. Higher means the user looked away from fixation. |
| `fixationMethod`    | string | no       | Free-form description of how fixation was monitored — e.g. `"blind-spot"`, `"head-tracker"`, `"gaze"`, `"none"`. |

### 3.7a `reliabilityIndices`

`reliabilityIndices` is an OPTIONAL top-level object carrying the raw integer counts needed to compute the FA% and FPRR% metrics defined by Dzwiniel et al. 2017. When present, **all four fields are REQUIRED**.

| Field                      | Type    | Description                                                                                            |
| -------------------------- | ------- | ------------------------------------------------------------------------------------------------------ |
| `catchTrialsPresented`     | integer | Number of blindspot catch trials presented during the test. MUST be ≥ 0.                               |
| `catchTrialsFalsePositive` | integer | Catch trials the patient reported seeing (fixation-loss signal). MUST be ≥ 0.                          |
| `falsePositiveIsiPresses`  | integer | Key presses during stimulus-absent ISI windows (trigger-happy signal). MUST be ≥ 0.                    |
| `truePositiveResponses`    | integer | Valid detections on real (non-catch) stimulus presentations. MUST be ≥ 0.                              |

**Computed metrics** (derive at display time; do NOT store):

| Metric  | Formula                                                                                       | Normal range (Dzwiniel 2017, n=21 healthy controls) |
| ------- | --------------------------------------------------------------------------------------------- | --------------------------------------------------- |
| FA%     | `(catchTrialsPresented − catchTrialsFalsePositive) / catchTrialsPresented × 100`              | 79–99 %                                             |
| FPRR%   | `(catchTrialsFalsePositive + falsePositiveIsiPresses) / (catchTrialsFalsePositive + falsePositiveIsiPresses + truePositiveResponses) × 100` | 0.3–2.3 % |

Where:
- **FA% (Fixation Accuracy)** measures the percentage of catch trials where the patient did not respond (correctly held fixation). Low FA% indicates poor fixation.
- **FPRR% (False Positive Response Rate)** measures the proportion of button presses that were not triggered by a real stimulus. High FPRR% indicates a trigger-happy response pattern.

**Example**:

```json
"reliabilityIndices": {
  "catchTrialsPresented": 10,
  "catchTrialsFalsePositive": 1,
  "falsePositiveIsiPresses": 2,
  "truePositiveResponses": 87
}
```

This gives FA% = (10 − 1) / 10 × 100 = 90% (normal), FPRR% = (1 + 2) / (1 + 2 + 87) × 100 ≈ 3.3% (slightly elevated).

Producers SHOULD emit `reliabilityIndices` whenever the test monitors catch trials. The four raw counts MUST be integers ≥ 0; any value of `catchTrialsPresented = 0` makes FA% undefined — consumers MUST handle this gracefully (e.g. display "N/A").

### 3.8 `isopters`

`isopters` is an OPTIONAL array of precomputed per-stimulus area summaries. Including this array is purely a convenience; consumers SHOULD always be able to recompute these values from `points`.

| Field         | Type   | Required | Description                                                     |
| ------------- | ------ | -------- | --------------------------------------------------------------- |
| `stimulusKey` | string | yes      | Reference to a `key` in the `stimuli` array.                    |
| `areaSqDeg`   | number | yes      | The area enclosed by the detected boundary, in square degrees.  |

### 3.9 `software`

| Field     | Type   | Required | Description                             |
| --------- | ------ | -------- | --------------------------------------- |
| `name`    | string | yes      | Producing application name.              |
| `version` | string | yes      | Producing application version.           |
| `url`     | string | no       | Homepage of the producing application.   |

## 4. Test types

### 4.1 Kinetic (`"kinetic"`)

A kinetic test moves a stimulus from outside the visual field inward along each meridian until the user signals detection. Points typically have `detected = true`; points with `detected = false` represent meridians where the stimulus reached the central area without being seen.

`responseTimeMs` and `rawEccentricityDeg` SHOULD be present for kinetic points when reaction-time compensation is applied. `sensitivityDb` is NOT meaningful for pure kinetic perimetry and SHOULD be omitted.

### 4.2 Static (`"static"`)

A static test flashes individual stimuli at fixed grid positions. Each point represents one trial; `detected = false` means the stimulus was not perceived in the response window. For threshold-based static perimetry (e.g. SITA), `sensitivityDb` SHOULD be present on every point.

### 4.3 Ring (`"ring"`)

A ring/arc test maps scotoma boundaries via user-controlled arc expansion. `sectorIdx` SHOULD be present so consumers can group points by sector.

## 5. Coordinate system

OVFX uses a single canonical coordinate system:

- **Origin** is the fixation point.
- **`meridianDeg`** is the angle in degrees, counter-clockwise from the positive x-axis. `0°` is to the user's right, `90°` is up.
- **`eccentricityDeg`** is the radial distance from fixation in degrees of visual angle.
- **Eye orientation:** the data is recorded **as the user sees it**. The temporal field of the right eye is on the user's right (positive x).

Screen-specific terms (pixels, fixation offset) are confined to `calibration.setup.screen` and never leak into `points`.

## 6. Extensions

### 6.1 Top-level `extensions` object

The top-level `extensions` object MAY contain implementation-defined fields. Each top-level key MUST be a reverse-DNS namespace owned by the implementer (e.g. `"com.example.myapp"`).

### 6.2 Underscore-prefixed point fields

Per-point fields whose name begins with an underscore MUST be preserved by tools that round-trip the document and MUST NOT be interpreted by general-purpose consumers.

Tools that re-emit an OVFX document **MUST preserve** unknown top-level fields, unknown `extensions` keys, and underscore-prefixed point fields.

## 7. Conformance

A **conforming OVFX document** is a JSON document that:

1. Validates against the JSON Schema published at `schema/v0.4.0/ovfx.schema.json`.
2. Contains all REQUIRED fields enumerated in §3.
3. Uses only the test types defined in §4 (or an extension type explicitly negotiated between producer and consumer).
4. Uses the coordinate system defined in §5.
5. If `calibration.setup.type === "screen"`, includes a `calibration.setup.screen` sub-object with `maxEccentricityDeg`.

A consumer **MUST** reject documents whose `ovfxVersion` is from a higher MAJOR version than it supports.

## 8. Versioning

- **MAJOR** — backward-incompatible schema change.
- **MINOR** — backward-compatible additions.
- **PATCH** — clarifications and editorial corrections.

Each spec version lives at a stable URL under `spec/<version>.md` and `schema/<version>/`. Published spec files are immutable except for editorial errata, which are recorded in `CHANGELOG.md`.

## 9. Security and privacy considerations

- OVFX documents **MUST NOT** contain PHI by default.
- A self-test app emitting OVFX SHOULD prompt the user before including `subject.anonymousId`.
- OVFX documents are plain JSON and are not encrypted by the format.
- The `notes` field is free-form and is the most likely place for a user to accidentally enter PHI. Implementers SHOULD warn users before allowing free-form notes.

## 10. Migration from 0.3.0

0.4.0 is a **purely additive** release. Every 0.3.0 document is a valid 0.4.0 document after bumping `ovfxVersion` from `"0.3.0"` to `"0.4.0"`. No field is renamed, removed, or narrowed.

0.4.0 adds the following OPTIONAL fields, motivated by cross-device comparability of threshold and response-time measurements:

- `test.stimulusDurationMs` — presentation duration of each stimulus.
- `test.interStimulusIntervalMs` — fixed minimum interval between stimuli.
- `test.interStimulusIntervalJitterMs` — additional random interval.
- `stimuli[].luminanceCdM2` — absolute luminance of a stimulus, in cd/m².
- `calibration.setup.screen.maxLuminanceCdM2` — the 0 dB reference luminance of the producing screen.
- `calibration.setup.screen.backgroundLuminanceCdM2` — background luminance of the screen during the test (matching the existing `calibration.setup.bowl.backgroundLuminanceCdM2`).
- `points[].catchTrial` (optional boolean) — marks a point as a blindspot catch trial. Catch-trial points MUST be excluded from isopter and sensitivity aggregations by consumers. See §3.6a.
- `reliabilityIndices` (optional top-level object) — raw integer counts for FA% and FPRR% computation, following the nomenclature of Dzwiniel et al. 2017 (PLoS ONE 12(10):e0186224). When present, all four fields (`catchTrialsPresented`, `catchTrialsFalsePositive`, `falsePositiveIsiPresses`, `truePositiveResponses`) are required. See §3.7a.

Producers that do not run a photometric calibration MAY omit every `*CdM2` field; `sensitivityDb` values in such documents remain device-relative, as in 0.3.0.

### Migration from 0.2.0

Unchanged from 0.3.0: move `calibration.maxEccentricityDeg` into `calibration.setup.screen.maxEccentricityDeg` (for screen tests) and bump `ovfxVersion`. See `spec/v0.3.0.md` §10 for the full procedure.
